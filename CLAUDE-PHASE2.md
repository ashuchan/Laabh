# CLAUDE-PHASE2.md — Paper Trading Engine & Analyst Scoreboard

## Overview
Phase 2 adds the **paper trading engine** and **analyst performance tracking** on top
of the data collection pipeline built in Phase 1. Users can execute virtual trades,
track portfolio P&L against benchmarks, and see which analysts actually produce
profitable signals.

## Prerequisites
- Phase 1 fully functional (data collection, signal extraction, price feeds)
- Database populated with instruments, sources, and initial price data
- At least 1 week of signal data for analyst scoring to be meaningful

## New Components

### 1. Paper Trading Engine (`src/trading/`)
```
src/trading/
├── __init__.py
├── engine.py              # Core trade execution logic
├── order_book.py          # Order management (market, limit, stop-loss)
├── portfolio_manager.py   # Portfolio valuation, P&L calculation
├── risk_manager.py        # Position sizing, max allocation checks
└── benchmark.py           # Nifty 50 / Sensex benchmark comparison
```

#### Trading Engine Rules
- **Market orders**: Execute immediately at current LTP from `price_ticks`
- **Limit orders**: Store in `pending_orders` table, check against each new tick
  - BUY LIMIT: execute when LTP <= limit_price
  - SELL LIMIT: execute when LTP >= limit_price
- **Stop-loss orders**: 
  - BUY SL: execute when LTP >= trigger_price
  - SELL SL: execute when LTP <= trigger_price (protective stop)
- **Brokerage simulation**: Deduct realistic charges
  - Brokerage: 0.03% or ₹20 per order (whichever is lower)
  - STT: 0.1% on sell side (delivery), 0.025% (intraday)
  - Transaction charges: 0.00345% (NSE)
  - GST: 18% on brokerage
  - Stamp duty: 0.015% on buy side
- **Position sizing**: Never allow a single stock to exceed 10% of portfolio (configurable)
- **Short selling**: Not supported in Phase 2 (delivery-only model)

#### Trade Execution Flow
1. User submits order (via API or signal-triggered auto-trade)
2. `risk_manager.validate()` checks: sufficient cash, position limits, market hours
3. If market order: execute immediately at LTP, deduct charges
4. If limit/SL: store in `pending_orders`, `order_book` checks on each tick
5. On execution: update `trades`, `holdings`, `portfolios` tables
6. Create notification: "BUY 50 RELIANCE @ ₹2,456.30 — ₹1,22,815 invested"
7. Recalculate portfolio metrics (total P&L, day P&L, weight distribution)

#### Portfolio Valuation (runs every 5 minutes during market hours)
1. For each holding: current_value = quantity × current LTP
2. P&L = current_value - invested_value (per holding and total)
3. Day P&L = current_value - (quantity × previous close)
4. Store daily snapshot at market close → `portfolio_snapshots`
5. Compare against benchmark: Nifty 50 return since portfolio creation date

### 2. Analyst Scoreboard (`src/analytics/`)
```
src/analytics/
├── __init__.py
├── analyst_tracker.py     # Match signals to analysts, track outcomes
├── signal_resolver.py     # Check if signals hit target/SL/expired
├── source_scorer.py       # Score data sources by signal quality
├── auto_trader.py         # Meta-paper-trade every signal automatically
└── reports.py             # Generate daily/weekly performance reports
```

#### Signal Resolution Logic (runs every hour during market hours + at close)
For each signal with `status = 'active'`:
1. Fetch current price of the instrument
2. If signal.action = 'BUY':
   - If current_price >= target_price → status = 'hit_target', calc P&L%
   - If current_price <= stop_loss → status = 'hit_stoploss', calc P&L%
   - If signal.expiry_date < now() → status = 'expired'
3. If signal.action = 'SELL':
   - Inverse logic for target and stop-loss
4. On resolution: update signal, update analyst stats, create notification

#### Analyst Scoring Algorithm
Run nightly at 6 PM IST via `update_analyst_scores()` stored procedure + Python enrichment:
```
credibility_score = (
    hit_rate × 0.40 +                    # what % of calls hit target
    normalized_avg_return × 0.25 +        # avg return per signal (capped at 10%)
    consistency × 0.20 +                  # low variance in returns
    recency_weight × 0.15                 # recent signals weighted more
)
```
- Minimum 5 resolved signals before scoring (below = default 0.5)
- Decay factor: signals older than 90 days contribute 50% weight
- Track per-sector accuracy: analyst may be great for IT but bad for pharma

#### Auto-Trader (Meta-Paper-Trading)
For EVERY signal generated by the system:
1. Create an entry in `signal_auto_trades`
2. Record entry_price = current LTP at signal time
3. Use target_price and stop_loss from the signal
4. Resolve using same logic as signal_resolver
5. Aggregate results by source and analyst
6. This data feeds back into credibility_score and source prioritization

### 3. Pending Orders Table (add to schema)
```sql
CREATE TABLE pending_orders (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    portfolio_id    UUID NOT NULL REFERENCES portfolios(id),
    instrument_id   UUID NOT NULL REFERENCES instruments(id),
    signal_id       UUID REFERENCES signals(id),
    trade_type      trade_type NOT NULL,
    order_type      order_type NOT NULL,
    quantity        INT NOT NULL,
    limit_price     NUMERIC(12,2),
    trigger_price   NUMERIC(12,2),
    status          VARCHAR(20) DEFAULT 'pending',  -- pending, executed, cancelled, expired
    valid_till      TIMESTAMPTZ,                    -- GTD (good till date)
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    executed_at     TIMESTAMPTZ,
    cancelled_at    TIMESTAMPTZ
);
```

## Scheduler Updates
Add these jobs to the APScheduler from Phase 1:
- `resolve_signals` — every 30 min during market hours (9:15-15:30 IST)
- `update_portfolio_values` — every 5 min during market hours
- `check_pending_orders` — every 1 min during market hours (fast)
- `daily_snapshot` — once at 15:35 IST (after market close)
- `update_analyst_scores` — once at 18:00 IST
- `generate_daily_report` — once at 18:30 IST (Telegram summary)
- `auto_trade_signals` — triggered on each new signal creation (event-driven)

## Rules for Claude Code
- All rules from Phase 1 CLAUDE.md still apply
- Trading engine must be deterministic: same inputs → same outputs
- All monetary calculations use `Decimal` (not float) to avoid rounding errors
- Portfolio updates must be atomic (use DB transactions)
- Never execute trades outside market hours (9:15-15:30 IST, Mon-Fri)
- Check NSE holiday calendar before executing (store holidays in system_config)
- API responses must include proper error messages, not just 500 errors
- Rate limit the API: max 60 requests/minute (protect against accidental loops)
